REST & SaaS connectors

Projects under Connectors/<Category>/<Vendor>/ implement vendor REST (or HTTP) APIs as IDataSource instances. They are not RDBMS engines: entities map to API resources and command-annotated operations, often backed by TheTechIdea.Beep.WebAPI.

Standard pattern (WebAPIDataSource)

  • Base type: Most connectors derive from WebAPIDataSource (namespace TheTechIdea.Beep.WebAPI), which centralizes HTTP, auth scaffolding, and editor integration.
  • Registration: [AddinAttribute(Category = DatasourceCategory.Connector, DatasourceType = DataSourceType.<Vendor>)] on the concrete datasource class.
  • Connection: WebAPIConnectionProperties is applied when the connection is not already typed; base URL, headers, and OAuth/API-key fields vary by vendor.
  • Entities: Logical names (e.g. tweets.search, users.by_id) are listed in EntitiesNames / Entities and map to endpoint templates or route fragments.
  • Commands: Operations exposed to the editor/runtime use CommandAttribute (and related metadata) so CRUD-like calls route to the correct HTTP method and path.
  • Models: DTOs and enums live in Models.cs (or a Models folder) under namespace TheTechIdea.Beep.Connectors.<Category>.<Service>.Models.

Refactoring inventory and conventions: Help/EXECUTION_SUMMARY.md (open from repo; raw markdown in browser may be plain text). Rough scale: 105+ connectors analyzed, high coverage of CommandAttribute and Models.

How flagship provider pages document APIs

Each conn-*.html file under Help/providers/ documents that connector’s command surface: public methods decorated with CommandAttribute, with Name, Caption, and ObjectType when the source sets them (some connectors omit Name; the editor then keys off the method name). A Model types subsection lists the main DTO classes and namespace; Connection (or auth) is typically last before Also read. Cross-cutting IDataSource behavior is described in IDataSource. The master index with hubs and DataSourceType is in Flagship provider pages (phase 09) below.
